# 求职地图 V0.4 验收 Checklist

## 1. 使用方式

本 checklist 用于 V0.4 开发、测试和上线前验收。

每项建议标记：

- PASS：通过。
- FAIL：未通过。
- N/A：本轮不适用。

P0 项全部 PASS 后，才算 V0.4 核心可交付。

## 1.1 2026-07-08 当前自动化验收状态

当前 V0.4 已经具备“求职诊断智能体工作流雏形”的自动化验收基线。以下项目已由测试覆盖，不再只依赖手工检查：

- PASS：有 JD 模式产品级 QA。真实感样本包含简历、目标 JD、求职困惑，报告必须输出岗位匹配、5 个面试追问、7 天行动计划，并绑定已确认经历证据。
- PASS：无 JD 模式产品级 QA。真实感样本包含简历和求职困惑但没有 JD，报告必须输出 2-3 个可搜索岗位方向，每个方向包含 7 天验证动作，并且不得伪装成 JD 匹配。
- PASS：经历资产卡证据边界。后续报告只读取确认使用或编辑后确认的经历，待确认、暂不使用、空白经历不会进入报告证据。
- PASS：动态追问契约。用户界面不展示 HR、TAR、PART、PREP、为什么问、事实回忆维度等内部标签。
- PASS：真实 AI 报告质量失败兜底。结构合法但存在 quality blocker 的深度报告不会直接展示给用户，会降级为保守基础版报告。
- PASS：用户端隐藏模型诊断细节。浏览器响应不展示 token、成本、模型名、usage、moduleUsages 或内部 modules 原文。
- PASS：基础版报告兜底。深度模型失败时仍能返回保守、非空、可行动的基础版报告。

自动化验证入口：

- `npm.cmd test`：当前覆盖 10 个测试文件、121 个测试。
- `npm.cmd run build`：当前覆盖 TypeScript 与生产构建。
- 产品级 QA 重点文件：`server/productQa.test.ts`、`server/productQa.ts`。

仍建议在 V0.4 结束前手工抽查：

- 手机端完整走一遍有 JD / 无 JD 两条路径。
- 真实 API Key 下各跑一次有 JD / 无 JD 样本，确认模型输出不会过度空泛。
- 反馈区真实提交一次，确认用户可以表达“有用 / 不准 / 愿不愿付费”等信息。

- [ ] 用户界面不展示为什么问。

# 求职地图 V0.4 PRD 草稿

## 1. 版本目标

V0.4 的总目标是：多模型稳定交付 + 高质量求职行动报告版。

本版本不追求让 AI 每次一次性完美生成，而是要求：

- 模型可以失败，但用户一定能拿到结果。
- 报告必须可信、可用、可行动。
- 所有判断基于用户确认过的真实经历。
- 不诱导造假，不夸大，不制造焦虑。
- 手机端和电脑端都能完整跑完流程。

V0.4 要从“AI 求职诊断工具”升级为“稳定交付求职行动报告的产品流程”。

## 2. 核心用户问题

### 2.1 AI 不稳定导致流程中断

DeepSeek 可能失败、超时或输出结构不合格。如果用户因此拿不到报告，会直接破坏信任。

V0.4 要解决的问题是：AI 深度内容可以延迟或降级，但用户不能被卡死。

### 2.2 报告可信度不足

如果 AI 根据简历自由推断、夸大经历、补写数据，报告看起来会更完整，但用户无法真正用于投递和面试。

V0.4 要解决的问题是：报告必须保守可信，宁可标注证据不足，也不能替用户编造。

### 2.3 经历未确认就进入分析

AI 初步识别的经历可能存在分类错误、遗漏、时间不准、职责夸大等问题。

V0.4 要解决的问题是：用户必须先核对经历资产卡，后续诊断只基于用户确认后的信息。

### 2.4 用户不知道如何回忆经历证据

很多用户不是没有经历，而是不知道 HR 会关心什么，也不知道如何从任务、行动、结果、复盘中提取证据。

V0.4 要解决的问题是：动态追问要像温和的 HR/面试教练，帮助用户回忆真实事实，而不是给用户可照抄的虚构答案。

追问方法可以使用 HR 视角、TAR、PART、PREP 等结构，但这些是系统内部生成逻辑，不应作为方法名、推理链或“挖掘套路”展示给用户。

### 2.5 无 JD 时方向建议太泛

用户没有目标 JD 时，常见输出容易变成“适合运营、产品、市场”等泛泛建议。

V0.4 要解决的问题是：无 JD 模式必须给出可用真实 JD 验证的岗位探索入口。

### 2.6 有 JD 时匹配分析容易误伤用户

匹配不足不等于用户能力差，很多时候只是材料无法证明能力。

V0.4 要解决的问题是：有 JD 模式要区分“现有材料是否能证明岗位要求”和“用户本人是否具备能力”。

### 2.7 手机端复杂报告难以阅读

表格、长文本、固定底栏和错误状态容易在手机端造成横向溢出或操作中断。

V0.4 要解决的问题是：移动端必须能完整跑完输入、确认、追问、报告查看和失败后继续。

## 3. 产品流程范围

### 3.1 开始页

保留 V0.3 的双模式入口：

- 用户阶段：准应届生 / 应届生。
- 诊断模式：无 JD 经历盘点 / 有 JD 定制诊断。

页面要求：

- 首屏直接进入诊断流程，不做营销型 landing page。
- 显示隐私提示、真实性声明、真实 AI 状态。
- 文案温和、克制，不制造求职焦虑。

### 3.2 简历与基础信息输入

支持用户输入：

- 简历文本。
- 学历、学校、专业、毕业时间、城市。
- 目标方向。
- 经历材料。
- 有 JD 模式下的目标岗位 JD。

文件上传仍可作为占位能力时，必须明确说明暂不承诺完整解析能力。

有 JD 模式规则：

- 用户在开始页选择有 JD 模式后，必须先补充目标岗位 JD。
- JD 不是只服务后面的匹配分析，也要作为动态追问的前置输入。
- 系统需要先将 JD 拆成岗位要求、职责、工具、经验、软素质等结构化要求。
- 经历资产卡确认完成后，动态追问必须同时结合结构化岗位要求和用户已确认经历。
- 如果 JD 过长，按 Kimi 触发规则先做结构化摘录。

### 3.3 经历资产卡确认

AI 初步识别后，用户必须核对经历卡。

经历卡类型包括：

- 教育。
- 实习。
- 项目。
- 校园。
- 兼职。
- 荣誉。
- 技能作品。
- 缺口 / 补强卡。

卡片状态包括：

- 待确认。
- 确认使用。
- 编辑后确认。
- 暂不使用。

规则：

- 所有经历卡必须被用户处理后才能进入下一步。
- 未确认卡存在时，点击下一步要提示并定位到对应卡片。
- 只有“确认使用”和“编辑后确认”的经历卡进入后续分析。
- 后续诊断、改写、匹配、行动计划都不得使用未确认经历。

确认页提示文案建议：

> 请先核对你的经历卡。AI 已完成初步整理，但可能存在识别错误、遗漏或分类不准。请你确认、修改或删除不准确的内容。后续诊断会以你确认后的信息为准。这一步不是考试，只是为了让后面的分析更贴近你的真实情况。

### 3.4 动态追问

动态追问的目标是帮助用户发现真实经历里的求职证据。

追问不是为了让用户“包装自己”，而是帮助用户把已经发生过的事情说清楚。

#### 3.4.1 追问输入

动态追问必须结合：

- 用户已确认经历。
- 用户已有补充回答。
- 用户阶段和目标方向。
- 有 JD 模式下的岗位 JD 要求。

有 JD 模式下的顺序要求：

1. 用户先补充目标岗位 JD。
2. 系统结构化 JD 岗位要求。
3. AI 初步识别用户经历资产卡。
4. 用户确认经历卡。
5. 系统再结合 JD 要求和已确认经历生成动态追问。

有 JD 模式下，每个追问问题必须绑定：

- 岗位要求。
- 用户经历。
- 为什么问。
- 事实回忆维度。

#### 3.4.2 追问方法库

系统可以使用以下方法生成追问和帮助用户总结，但不能诱导用户编造。

HR 视角：

- 这个经历能证明什么能力？
- 面试官可能追问哪里？
- 这个表述是否有证据支撑？
- 用户是否能解释自己的具体贡献？

TAR 原则：

- Task：当时要完成什么任务？
- Action：你具体做了什么？
- Result：最后产生了什么结果？

PART 原则：

- Problem：当时遇到什么问题？
- Action：你采取了什么行动？
- Result：产生了什么结果？
- Thinking：你后来有什么复盘或理解？

PREP 法则：

- Point：你想表达的能力点是什么？
- Reason：为什么这个经历能证明它？
- Example：有哪些真实细节可以支撑？
- Point：最后如何回到岗位要求或能力点？

使用边界：

- 这些方法用于提问、拆解和总结结构。
- 这些方法同样适用于无 JD 和有 JD 两种模式。
- 有 JD 模式下，这些方法必须围绕岗位要求和用户确认经历使用。
- 无 JD 模式下，这些方法围绕用户确认经历、目标方向和可迁移能力使用。
- HR 视角、TAR、PART、PREP、为什么问、事实回忆维度属于内部生成依据，不在用户界面直接展示。
- 用户侧只展示自然、具体、可回答的问题，以及必要的简短说明。
- 不给完整示例答案。
- 不展示“可能挖出的亮点”。
- 不给“你可以这样回答”的明确答案。
- 不暗示用户添加没有发生过的数据、角色、结果。

#### 3.4.3 追问输出要求

每轮追问：

- 1-3 个问题。
- 问题具体，不抽象。
- 可跳过、稍后补充、快速完成。
- 可以给事实型鼓励，但不能鸡汤化。
- 不展示 HR/TAR/PART/PREP 等方法标签。
- 不展示“为什么这样问”的内部逻辑。

问题示例方向：

- 你当时负责的是哪一部分？
- 这个任务持续了多久？
- 你用了哪些工具或方法？
- 有没有可以确认的结果、反馈、数量、交付物？
- 如果面试官问“这件事主要难在哪里”，你会想到什么？

### 3.5 无 JD 模式：可探索岗位方向

模块名称：可探索岗位方向。

定位：

- 不替用户决定职业方向。
- 给用户可用真实 JD 验证的岗位探索入口。

每个方向必须包含：

- 方向名称。
- 可搜索岗位名称 3-5 个。
- 为什么可以探索。
- 对应经历证据。
- 当前缺口。
- 探索优先级。
- 7 天验证动作。

规则：

- 每个方向必须绑定至少一段已确认经历。
- 不允许证据错位。
- 不输出现实中难以搜索的抽象岗位。
- 不使用“最适合、不适合、强烈推荐”等绝对词。

### 3.6 有 JD 模式：岗位要求匹配分析

模块标题：岗位要求匹配分析。

副标题：

> 以下分析基于你确认过的经历和目标岗位 JD，用于判断当前材料和岗位要求的对应关系。匹配程度不是对你的能力评价，而是看现有材料是否能证明相关能力。

匹配程度四档：

- 匹配较强。
- 有一定匹配。
- 需要补充证据。
- 当前证据不足。

投递判断四档：

- 建议优先投递。
- 可以投递，建议先优化简历。
- 可以作为尝试方向。
- 建议先补强后再重点投递。

规则：

- 每条岗位要求必须绑定已确认经历，或标注当前证据不足。
- 不允许证据错位。
- 不把“材料证据不足”写成“用户能力不行”。
- 手机端使用卡片，不使用横向表格。

### 3.7 简历改写建议

模块名：简历改写建议。

原则：

- 只优化表达，不新增事实。
- 只翻译能力，不伪造经历。
- 不使用“增强版改写”。
- 不把参与、协助包装成负责、主导。
- 不把课程作业包装成企业真实项目。

无 JD 结构：

- 对应经历。
- 原表达问题。
- 可体现能力。
- 可直接使用版。
- 补充信息后可用版。
- 使用提醒。

有 JD 结构：

- 回应的岗位要求。
- 对应你的经历。
- 原表达问题。
- 为什么这样改。
- 可直接使用版。
- 补充信息后可用版。
- 使用提醒。

数量要求：

- 无 JD 至少 3 条。
- 有 JD 至少 3 条。

### 3.8 面试追问与回答准备

模块名：面试追问与回答准备。

适用范围：

- 只服务有 JD 模式。
- 无 JD 模式不做“面试可讲故事”。

必须输出 5 个面试问题。

每个问题包含：

- 面试问题。
- HR 为什么可能会问。
- 关联岗位要求。
- 可使用的真实经历。
- 回答思路。
- 占位式表达。
- 注意边界。

规则：

- 不输出可直接照抄的虚构完整答案。
- 回答思路可以使用 TAR、PART、PREP 等结构。
- 占位式表达必须提醒用户替换为真实信息。

### 3.9 下一步行动计划

模块名：下一步行动计划。

两种模式都需要。

固定阶段：

- 7 天内。
- 14 天内。
- 30 天内。

每个阶段至少 2 条行动。

每条行动包含：

- 要做什么。
- 为什么做。
- 怎么做。
- 完成标准。
- 对求职有什么帮助。

无 JD 聚焦：

- 方向验证。
- 整理经历。
- 补材料。
- 搜索真实 JD。

有 JD 聚焦：

- 投递前优化。
- 补岗位证据。
- 准备面试。
- 调整简历表达。

### 3.10 反馈区与数据处理

保留 V0.3 反馈区：

- 报告是否有帮助。
- 哪部分最有用。
- 哪部分不准。
- 是否愿意继续使用。
- 是否匿名授权用于产品优化。

规则：

- 不默认公开案例。
- 不默认用于训练。
- 保留清除本次分析数据入口。

## 4. 多模型策略

### 4.1 模型职责

DeepSeek：

- 全流程主模型。
- 负责主要结构化分析、诊断、报告生成。

Qwen：

- 当前模块备用模型。
- DeepSeek 失败、超时或 schema 不合格时，Qwen 接手当前模块。
- Qwen 不继续 DeepSeek 半成品。
- Qwen 拿同一份完整 AI Task Package，从头完成当前模块，并输出同样结构。

Kimi：

- 只做长文本结构化摘录。
- 不做判断、推荐、改写。
- 默认不调用。

规则模板：

- 最终兜底。
- 生成基础版报告。

### 4.2 Kimi 调用规则

默认不调用 Kimi。

仅在以下情况触发：

- JD 超过约 5000 中文字。
- 用户材料超过约 8000 中文字。
- 任务包预计 token 超过安全阈值。
- DeepSeek 因上下文过长失败一次后。

Kimi 输出必须包含：

- 来源片段。
- 待核实信息。
- 结构化字段。

Kimi 输出不得包含：

- 岗位判断。
- 投递建议。
- 简历改写。
- 面试答案。
- 职业方向推荐。

### 4.3 AI Task Package

每个模块调用 AI 时，应使用完整 AI Task Package。

任务包至少包含：

- 用户阶段。
- 诊断模式。
- 基础信息。
- 已确认经历卡。
- 用户补充回答。
- JD 原文或 Kimi 摘录结构。
- 当前模块任务目标。
- 输出 schema。
- 安全红线。
- 禁止使用的未确认信息。

### 4.4 失败与兜底

模块级生成链路：

1. DeepSeek 生成当前模块。
2. 如果失败、超时或 schema 不合格，Qwen 用同一份完整任务包从头生成当前模块。
3. 如果 Qwen 仍失败，规则模板生成基础版内容。
4. 基础版内容展示后，系统可继续尝试补全深度内容。
5. 深度内容补全不得清空已有基础版内容。

## 5. 基础版报告

基础版报告不是失败报告，而是保守可信的最低可交付版本。

原则：

- 不空。
- 不乱。
- 不假。
- 可参考。
- 能行动。

页面说明建议：

> 当前已为你生成基础版报告。内容基于你确认过的信息和稳定规则整理，会偏保守，但不会替你编造经历。你可以先参考，系统也会继续尝试补全深度内容。

基础版报告至少包含：

- 已确认经历摘要。
- 当前可参考的方向或岗位匹配概览。
- 简历表达提醒。
- 7 天内行动建议。
- 需要补充的信息。
- 真实性与边界提示。

## 6. 用户端交互与移动端

### 6.1 失败态与重试

用户端要求：

- AI 失败后可重试。
- AI 失败后可使用基础版继续。
- 基础版生成后可继续尝试补全深度内容。
- 补全失败不清空基础版。
- 用户端隐藏 token、成本、模型明细。

### 6.2 等待态

每个耗时操作必须显示预计等待时间，例如：

- 正在整理经历卡，预计 20-40 秒。
- 正在生成报告，预计 1-2 分钟。
- 当前生成较慢，你可以稍后重试或先查看基础版。

### 6.3 移动端

移动端规则：

- 单栏布局。
- 表格改为卡片。
- 长文本不横向溢出。
- 按钮适合触控。
- 固定底栏不得遮挡正文。
- 所有报告模块可阅读、可复制、可继续。

## 7. 优先级

### 7.1 P0 必须做

- DeepSeek -> Qwen -> 规则模板 fallback。
- Qwen 使用完整 AI Task Package 从头生成当前模块。
- 经历资产卡强确认。
- 后续分析只使用已确认经历。
- 基础版报告。
- 有 JD 岗位要求匹配分析。
- 无 JD 可探索岗位方向。
- 动态追问绑定真实经历和事实回忆维度。
- 简历改写建议至少 3 条。
- 有 JD 面试追问固定 5 条。
- 下一步行动计划 7/14/30 天。
- 手机端完整跑通。

### 7.2 P1 应该做

- Kimi 长文本摘录触发机制。
- AI Task Package 标准化。
- schema 校验与模块级重试。
- 基础版后继续补全深度内容。
- 耗时操作预计等待时间。
- 失败态、重试态、继续查看基础版状态。
- 反馈区结构化记录。

### 7.3 P2 可以后置

- 成本、token、模型明细后台。
- 岗位证据库或 RAG。
- 支付能力。
- 多版本报告对比。
- 简历导出增强。
- 更复杂的岗位推荐排序算法。
- 用户历史数据沉淀与长期追踪。

## 8. 不做什么

V0.4 明确不做：

- 不让用户看到 token、成本、模型明细。
- 不把 Kimi 用作判断、推荐、改写模型。
- 不让 Qwen 续写 DeepSeek 的半成品。
- 不根据未经确认的经历做分析。
- 不编造学历、学校、公司、证书、时间、岗位名称、项目成果。
- 不把参与者写成负责人。
- 不把课程作业包装成企业真实项目。
- 不输出“最适合、不适合、强烈推荐”等绝对判断。
- 不输出可直接照抄的虚构面试答案。
- 不使用“增强版改写”。
- 无 JD 模式不做“面试可讲故事”。
- 不把基础版报告描述为失败报告。
- 不优先做商业化、支付、复杂后台。

## 9. 验收标准

### 9.1 流程验收

- 用户能完整走通无 JD 和有 JD 两条流程。
- 用户能完成输入、经历确认、动态追问、报告查看。
- 未处理所有经历卡时，不能进入下一步。
- 点击下一步时能提示并定位未确认卡片。
- 用户可跳过或稍后补充动态追问。

### 9.2 AI 稳定性验收

- DeepSeek 失败时，Qwen 能接手当前模块。
- Qwen 接手时，使用完整 AI Task Package 从头生成。
- Qwen 失败时，规则模板能生成基础版报告。
- 基础版报告展示后，深度补全失败不会清空已有内容。
- schema 不合格会触发重试或 fallback。

### 9.3 真实性验收

- 报告不使用未确认经历。
- 不新增用户没有提供或确认过的事实。
- 不把不确定信息写成确定事实。
- 不伪造数据、角色、成果、证书、公司、项目性质。
- 所有关键判断能追溯到已确认经历或 JD 要求。

### 9.4 无 JD 模式验收

- 输出“可探索岗位方向”模块。
- 每个方向包含 3-5 个可搜索岗位名称。
- 每个方向绑定至少一段已确认经历。
- 每个方向包含当前缺口和 7 天验证动作。
- 不输出抽象、无法搜索或绝对化岗位判断。

### 9.5 有 JD 模式验收

- 输出“岗位要求匹配分析”模块。
- 每条岗位要求绑定已确认经历，或标注当前证据不足。
- 输出匹配程度四档之一。
- 输出投递判断四档之一。
- 不把材料证据不足表达成用户能力不行。

### 9.6 动态追问验收

- 每轮追问 1-3 个问题。
- 每个问题绑定真实经历。
- 有 JD 时，每个问题绑定岗位要求。
- 问题能帮助用户回忆任务、行动、结果、复盘等事实。
- 内部可以使用 HR 视角、TAR、PART、PREP 等方法组织问题。
- 用户界面不展示 HR 视角、TAR、PART、PREP、为什么问、事实回忆维度或内部推理链。
- 不展示可能挖出的亮点。
- 不给完整示例答案。
- 不诱导用户编造。

### 9.7 简历改写验收

- 无 JD 至少 3 条。
- 有 JD 至少 3 条。
- 每条改写绑定对应经历。
- 有 JD 时每条改写绑定岗位要求。
- 改写只优化表达，不新增事实。
- 每条包含使用提醒。

### 9.8 面试模块验收

- 只在有 JD 模式出现。
- 固定输出 5 个面试问题。
- 每个问题包含 HR 为什么可能会问、关联岗位要求、可使用真实经历、回答思路、占位式表达、注意边界。
- 不输出虚构完整答案。

### 9.9 行动计划验收

- 两种模式都有“下一步行动计划”。
- 固定 7 天内、14 天内、30 天内。
- 每个阶段至少 2 条行动。
- 每条行动包含要做什么、为什么做、怎么做、完成标准、对求职有什么帮助。

### 9.10 移动端验收

- 手机端可完成完整流程。
- 表格在手机端改为卡片。
- 长文本不横向溢出。
- 关键按钮不被遮挡。
- 报告模块可阅读、可复制、可继续。

## 10. 风险与依赖

### 10.1 主要风险

多模型输出结构不一致：

- 需要强 schema 校验。
- Qwen 必须使用同一任务包输出同样结构。

基础版报告质量过低：

- 基础版不能只是错误兜底提示。
- 必须有最低内容标准和行动建议。

经历确认增加用户负担：

- 需要用清楚文案解释确认的意义。
- 交互要支持快速确认、编辑后确认、暂不使用。

动态追问诱导风险：

- 方法库只能用于提问和总结结构。
- 不能输出完整示例答案或暗示编造。

无 JD 方向泛化：

- 必须要求可搜索岗位名称和经历证据绑定。

有 JD 匹配误伤用户：

- 文案必须强调这是材料证明程度，不是能力评价。

移动端报告复杂：

- 表格必须卡片化。
- 长文本必须可读。

### 10.2 关键依赖

- AI Task Package 结构。
- 各模块输出 schema。
- DeepSeek/Qwen/Kimi API 可用性。
- 模型超时与重试策略。
- 规则模板报告生成器。
- 经历资产卡状态管理。
- 报告模块数据来源追踪。
- 移动端 UI 适配。
- 前端失败态、重试态、基础版继续态。

## 11. V0.4 产品取舍

V0.4 优先做“稳定交付”和“可信报告”，暂不追求功能数量。

本版本最重要的信任感来自：

- 不会卡死。
- 不会瞎编。
- 不会打击用户。
- 能帮用户做下一步行动。

因此，V0.4 应优先完成多模型兜底、经历确认、基础版报告、JD/无 JD 两条报告主链路，再考虑支付、后台数据、复杂推荐算法或 RAG。

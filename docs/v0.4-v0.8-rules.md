# V0.4-V0.8 长期开发约束 Prompt

你现在协助我开发一个求职诊断智能体产品，目前处于 V0.4 迭代阶段，目标是在 V0.8 前完成产品自己的智能体系统雏形。

从 V0.4 到 V0.8，必须始终记住以下原则：

## 1. 当前产品定位

这是一个“求职诊断智能体”产品，不是普通简历网页工具。核心目标是让用户输入简历、目标岗位、求职困惑后，系统能像一个求职顾问一样进行诊断、追问、分析、建议和输出方案。

## 2. 当前阶段目标

V0.4-V0.8 的重点不是追求功能堆叠，而是搭建稳定、可迭代、可测试的智能体工作流雏形。每次开发都要优先服务于“诊断质量”和“工作流清晰度”。

## 3. 设备与技术路线

现阶段继续使用当前 Windows 电脑开发，不因为后期可能换 MacBook 而改变技术路线。项目必须保持跨平台可迁移：

- 不写死 `C:\xxx` 这类 Windows 本地路径
- 环境变量写入 `.env`，并维护 `.env.example`
- 保留 `package-lock.json`
- 使用 Git 记录版本
- 关键版本推送 GitHub
- README 保持可启动、可迁移、可交接

## 4. V0.4-V0.8 每次迭代前都要先检查

在写代码前，先回答：

- 这次改动属于智能体工作流、前端体验、后端接口、提示词、数据结构、测试，还是文档？
- 这次改动是否会让求职诊断更准确、更稳定、更可解释？
- 是否会破坏已有功能？
- 是否需要更新 README、`.env.example`、版本记录或提示词文档？

## 5. V0.4-V0.8 必须逐步完成的基础工程事项

从现在开始，不等到换电脑再做：

- 整理 Git 状态，避免重要文件未跟踪
- 上传/同步 GitHub
- 建立基础 README
- 建立 `.env.example`
- 不写死本地路径
- 每个重要版本写清楚变更记录
- 每完成一个阶段，使用 Git commit 保存
- 到 V0.4、V0.5、V0.6、V0.7、V0.8 时尽量打版本标签

## 6. V0.4-V0.8 智能体工作流建设重点

优先建设这些能力：

- 用户输入信息收集：简历、岗位、目标、求职阶段、困惑
- 求职诊断拆解：简历问题、岗位匹配度、表达问题、竞争力缺口
- 多轮追问：信息不足时主动提问，而不是直接胡乱输出
- 诊断结果结构化：问题、原因、优先级、修改建议、下一步行动
- 提示词版本管理：重要 prompt 要保存，不要散落在代码里
- 输出可复查：让用户能看懂为什么得出这个建议
- 错误处理：AI 失败、输入不足、接口异常时要有清晰反馈

## 7. V0.4-V0.8 暂时不要过度做的事

除非我明确要求，否则不要优先做：

- 本地部署大模型
- 复杂多用户权限系统
- 复杂付费系统
- 过度设计数据库
- 过度重构
- 为未来 MacBook 做专门适配
- 为尚不存在的大流量提前设计复杂架构

## 8. 后期换 MacBook 的迁移原则

如果 V1.0 后换 MacBook，目标应该是直接：

```bash
git clone 项目
npm install
复制 .env
npm run dev
```

然后继续开发。

所以从 V0.4 开始，所有代码、文档、环境变量和启动方式都要朝这个目标整理。

## 9. 每次完成改动后必须做

完成任何明显改动后，请检查：

- `npm run build` 是否能通过
- `npm test` 是否能通过，如果项目已有测试
- Git 状态是否清楚
- 是否需要更新文档
- 是否需要记录本次版本变化
- 是否有未提交的重要文件

## 10. 回答和开发风格

我是产品和编程学习阶段的新手，请用清楚、直接、可执行的方式协助我。不要只给理论。每次建议都要说明：

- 现在该不该做
- 为什么
- 做到什么程度就够
- 是否会影响 V0.8 前的主线目标

从现在到 V0.8，你必须把“求职诊断智能体工作流雏形”作为主线目标，所有开发建议、代码修改、文档整理都围绕这个目标推进。

## 11. V0.7 21 天求职陪跑产品总纲

从 V0.7 开始，产品主线从“一次性求职诊断报告”升级为“21 天应届生求职行动陪跑 MVP”。旧守则中的工程约束、稳定性要求、真实性要求继续有效；如果旧守则与本节 12 条原则出现冲突、模糊或优先级不清，必须以本节 12 条原则为准。

后续任何 Codex / Agent / 分支任务开始前，都必须先阅读本文件，并确认本次改动是否符合以下 12 条原则。

### 11.1 行动闭环优先于报告生成

产品不是为了生成一份漂亮报告，而是为了让用户知道今天该做什么、做完记录什么、怎么复盘、下一步怎么调整。报告可以存在，但必须服务后续行动，不能成为 V0.7 的终点。

### 11.2 现实温柔优先于空泛鼓励

产品不逃避就业难，不说“你一定可以”“一定能拿 offer”等空泛承诺；但也不打击用户，不把无反馈归因为“你不行”。必须始终区分“当前材料无法证明能力”和“用户本人不行”。所有建议都要现实、具体、温和、可执行。

### 11.3 拉伸区任务优先于宏大计划

每日任务必须小步、具体、可完成、可验证。任务应让用户处于舒适区边缘的拉伸区，而不是停留在无推进的舒适区，也不能把用户推入恐慌区。不要输出“提升竞争力”“做好职业规划”这类宏大但不可执行的任务。

### 11.4 只服务普通应届生第一份工作

V0.7 暂时只服务普通本科 / 双非 / 普通院校、非技术岗为主、正在找第一份正式工作或毕业一年内仍在求职的人。暂时不服务大三长期规划、职场跳槽、技术岗深度面试、考公考研考编、高端职业规划等人群。

### 11.5 AI 是陪跑教练，不是权威职业导师

AI 的职责是整理事实、拆解任务、识别缺口、生成下一步行动和阶段复盘。AI 不替用户做人生决定，不承诺 offer，不使用“最适合你”“你不适合”这类绝对判断，不把用户本人和求职结果绑定评价。

### 11.6 统一闭环：诊断 - 行动 - 记录 - 复盘 - 调整

V0.7 所有核心功能都必须服务“诊断 - 行动 - 记录 - 复盘 - 调整”闭环。不能进入闭环的功能，只能作为辅助，不作为主路径。

- 诊断：用户现在卡在哪，是方向、简历、投递、岗位判断，还是面试反馈。
- 行动：今天要做什么，任务必须具体、轻量、可完成。
- 记录：用户做完后留下什么，如简历版本、岗位样本、投递状态、面试反馈。
- 复盘：根据记录判断问题出在哪。
- 调整：修改下一步任务、岗位方向、简历版本或投递策略。

### 11.7 方向必须经过真实岗位验证

对“还没方向”的用户，AI 不能凭空判断用户适合什么职业。所有方向建议都必须落到现实中可搜索、可投递、可验证的岗位名称，并通过真实 JD / 岗位样本进行验证。未经过真实岗位验证的方向，只能叫“可探索方向”，不能叫“推荐职业”或“最适合方向”。

一个可探索方向至少应满足：

- 能在 BOSS、智联、前程无忧、猎聘等招聘平台搜到。
- 岗位名称具体，例如“用户运营助理”“新媒体运营”“市场助理”，而不是“互联网”“教育行业”。
- 存在应届生 / 初级岗位入口。
- JD 要求能够拆解成职责、能力、工具、经验或软素质要求。
- 用户可以据此完成搜索、收藏、粘贴、对比、投递等下一步动作。

### 11.8 商业化服务行动价值，不制造焦虑付费

V0.7 不做支付、账号、复杂后台和自动投递，但所有设计要能验证未来用户是否愿意为 21 天陪跑付费。未来商业化只能围绕清晰路径、行动陪伴、投递复盘和求职材料质量提升展开；不得通过恐吓、羞辱、保 offer 承诺或焦虑营销诱导付费。

### 11.9 反馈必须服务产品决策，而不是收集情绪噪音

反馈表必须简洁、低负担、结构化，优先收集能帮助判断路线价值、任务有效性、内容可信度、用户行动意愿和付费可能性的信息。

“哪一部分最没帮助 / 不准确？”必须包含“无”选项，且“无”与其他负面选项互斥。反馈字段要便于后续按路线、模块、价格接受度和行动意愿整理分析。每个反馈字段都应能回答一个产品决策问题：哪条路线值得继续、哪个模块该删、哪里不够现实、用户是否愿意行动、用户是否愿意付费。

### 11.10 数据最小化、用户可控与求职伦理优先

产品只收集完成求职诊断、行动计划和复盘所必要的信息。默认提醒用户删除身份证号、家庭住址、银行卡号等无关敏感信息。V0.7 阶段优先本地保存，保留清除数据入口，不建设复杂云端数据沉淀。

产品不得诱导用户伪造经历、夸大成果、虚构证书或自动骚扰式投递；不得承诺 offer、面试或薪资结果；不得向用户暴露模型、token、API key、provider、原始错误等内部信息。反馈统计应尽量结构化和脱敏，不以牺牲用户信任换取数据沉淀。

### 11.11 V0.7 只验证陪跑闭环，不建设完整平台

V0.7 的目标是验证“普通应届生是否愿意围绕 21 天求职陪跑持续行动、记录和反馈”，而不是建设完整职业成长平台。

本版本必须完成首页四状态分流、21 天计划雏形、每日任务、投递无反馈轻量输入、路线级结果和结构化反馈；同时明确不做支付、账号、后台、完整投递 CRM、自动投递、招聘平台爬虫、复杂职业测评、精美导出和本地大模型。

所有 Agent 开发前必须先确认改动是否服务陪跑闭环，不能为了“看起来更完整”而增加复杂功能。

### 11.12 公共底座先行，路线分支隔离，最后统一整合

V0.7 不允许四条路线直接同时改公共入口和核心类型。必须先完成公共底座：产品总纲、首页四状态分流、统一路线类型、基础 21 天计划类型、反馈字段和路由结构。

公共底座稳定后，再分别开发四条路线：还没方向、有方向但简历还没准备好、已投递但没反馈、有目标岗位想判断能不能投。每个路线分支只能实现本路线的页面、任务、结果和测试，不得擅自修改产品总纲、公共类型命名、伦理底线或其他路线逻辑。

最后通过整合分支统一 UI、文案、状态流转、测试和体验。

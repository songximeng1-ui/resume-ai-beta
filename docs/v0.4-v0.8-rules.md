# V0.4-V0.8 长期开发约束 Prompt

你现在协助我开发一个求职诊断智能体产品，目前处于 V0.4 迭代阶段，目标是在 V0.8 前完成产品自己的智能体系统雏形。

从 V0.4 到 V0.8，必须始终记住以下原则：

## 1. 当前产品定位

这是一个“求职诊断智能体”产品，不是普通简历网页工具。核心目标是让用户输入简历、目标岗位、求职困惑后，系统能像一个求职顾问一样进行诊断、追问、分析、建议和输出方案。

## 2. 当前阶段目标

V0.4-V0.8 的重点不是追求功能堆叠，而是搭建稳定、可迭代、可测试的智能体工作流雏形。每次开发都要优先服务于“诊断质量”和“工作流清晰度”。

## 3. 设备与技术路线

现阶段继续使用当前 Windows 电脑开发，不因为后期可能换 MacBook 而改变技术路线。项目必须保持跨平台可迁移：

- 不写死 `C:\xxx` 这类 Windows 本地路径
- 环境变量写入 `.env`，并维护 `.env.example`
- 保留 `package-lock.json`
- 使用 Git 记录版本
- 关键版本推送 GitHub
- README 保持可启动、可迁移、可交接

## 4. V0.4-V0.8 每次迭代前都要先检查

在写代码前，先回答：

- 这次改动属于智能体工作流、前端体验、后端接口、提示词、数据结构、测试，还是文档？
- 这次改动是否会让求职诊断更准确、更稳定、更可解释？
- 是否会破坏已有功能？
- 是否需要更新 README、`.env.example`、版本记录或提示词文档？

## 5. V0.4-V0.8 必须逐步完成的基础工程事项

从现在开始，不等到换电脑再做：

- 整理 Git 状态，避免重要文件未跟踪
- 上传/同步 GitHub
- 建立基础 README
- 建立 `.env.example`
- 不写死本地路径
- 每个重要版本写清楚变更记录
- 每完成一个阶段，使用 Git commit 保存
- 到 V0.4、V0.5、V0.6、V0.7、V0.8 时尽量打版本标签

## 6. V0.4-V0.8 智能体工作流建设重点

优先建设这些能力：

- 用户输入信息收集：简历、岗位、目标、求职阶段、困惑
- 求职诊断拆解：简历问题、岗位匹配度、表达问题、竞争力缺口
- 多轮追问：信息不足时主动提问，而不是直接胡乱输出
- 诊断结果结构化：问题、原因、优先级、修改建议、下一步行动
- 提示词版本管理：重要 prompt 要保存，不要散落在代码里
- 输出可复查：让用户能看懂为什么得出这个建议
- 错误处理：AI 失败、输入不足、接口异常时要有清晰反馈

## 7. V0.4-V0.8 暂时不要过度做的事

除非我明确要求，否则不要优先做：

- 本地部署大模型
- 复杂多用户权限系统
- 复杂付费系统
- 过度设计数据库
- 过度重构
- 为未来 MacBook 做专门适配
- 为尚不存在的大流量提前设计复杂架构

## 8. 后期换 MacBook 的迁移原则

如果 V1.0 后换 MacBook，目标应该是直接：

```bash
git clone 项目
npm install
复制 .env
npm run dev
```

然后继续开发。

所以从 V0.4 开始，所有代码、文档、环境变量和启动方式都要朝这个目标整理。

## 9. 每次完成改动后必须做

完成任何明显改动后，请检查：

- `npm run build` 是否能通过
- `npm test` 是否能通过，如果项目已有测试
- Git 状态是否清楚
- 是否需要更新文档
- 是否需要记录本次版本变化
- 是否有未提交的重要文件

## 10. 回答和开发风格

我是产品和编程学习阶段的新手，请用清楚、直接、可执行的方式协助我。不要只给理论。每次建议都要说明：

- 现在该不该做
- 为什么
- 做到什么程度就够
- 是否会影响 V0.8 前的主线目标

从现在到 V0.8，你必须把“求职诊断智能体工作流雏形”作为主线目标，所有开发建议、代码修改、文档整理都围绕这个目标推进。

# V0.6 版本记录

## 2026-07-09：有 / 无岗位要求真实模型 smoke

改动类型：智能体工作流、后端接口、测试、文档。

本次在本地后端 real 模式下，分别对有岗位要求路线和无岗位要求路线各执行 3 次真实报告生成。测试样本覆盖实习、项目、校园、荣誉、技能作品和作品集资料卡，用于验证首轮自测修复后报告链路是否能进入真实模型并完成 V0.6 统计记录。

本次测试结果：

- 总运行次数：6。
- 技术可用率：100%（6/6 HTTP 200，均到达报告页）。
- 深度报告率：83.33%（5/6 `isBasic=false`）。
- 质量通过率：100%（6/6 `quality.passed=true`）。
- DeepSeek direct 模块级成功率：94.44%。
- Qwen fallback 触发次数：1。
- Qwen fallback 成功率：100%。
- 基础版兜底次数：1。
- 失败类型计数：`quality_blocker=1`，`parse=1`。

分路线观察：

- 有岗位要求路线 3 次均返回用户可见报告，其中 2 次为深度报告，1 次因最终质量检查进入基础版兜底；返回内容包含岗位要求匹配、改写建议、面试追问和行动计划。
- 无岗位要求路线 3 次均返回深度报告，其中 1 次触发 Qwen fallback 且成功；返回内容包含可探索方向、改写建议和行动计划，不输出岗位要求匹配或面试追问模块。

边界：

- 本次未打印或记录 `.env` 中的真实 API Key。
- 本次未新增环境变量，`.env.example` 暂无必要更新。
- 本次未修改提示词，提示词文档暂无必要更新。
- 本次未修改启动或部署方式，README 暂无必要更新。

## 2026-07-09：有岗位要求路线首轮自测修复

改动类型：智能体工作流、前端体验、提示词、数据结构、测试、文档。

本次根据 V0.6 私密内测首轮自测反馈，修复有岗位要求路线仍残留 V0.4 通用流程的问题。目标是让“简历 + 岗位要求”路线真正围绕岗位要求追问，并减少真实测试者在动态经历挖掘后的重复输入感。

本次调整：

- 用户可见文案把“目标岗位 JD”降噪为“目标岗位要求”或“岗位要求”。
- 动态经历挖掘不再只覆盖实习、项目、校园、兼职、荣誉，扩展到技能作品和作品集。
- 作品集从原先合并进技能作品，调整为独立经历资产卡。
- 有岗位要求路线的动态追问提示词明确要求结合岗位要求、当前经历和已有补充回答追问；技能作品和作品集必须围绕岗位要求中的工具、内容、数据、作品或交付物追问。
- 有岗位要求路线在最后一段动态追问后直接生成完整报告，不再进入二次“岗位要求证据匹配”输入页。
- 直接从动态追问页生成报告时，报告生成中的等待提示和失败提示会在当前页展示，避免用户误以为卡住。
- 报告页、反馈表和导出文案中可见的“JD 匹配分析 / JD 证据矩阵”调整为“岗位要求匹配分析 / 岗位要求证据矩阵”。

本次测试：

- 更新有岗位要求完整流程测试，验证技能作品和作品集都会进入动态追问。
- 更新有岗位要求路线测试，验证最后一段追问后直接生成完整报告，不再二次要求填写岗位要求。
- 更新报告生成失败测试，验证动态追问页直接生成报告失败时仍显示可重试错误。
- 更新无岗位要求路线测试，适配技能作品进入动态追问后的多段流程。

边界：

- 本次未新增环境变量，`.env.example` 暂无必要更新。
- 本次未修改部署方式，README 暂无必要更新。
- 本次修改了动态追问提示词，已在本记录中同步说明。
- 本次不新增支付、账号系统、历史报告库、复杂后台或数据库。

## 2026-07-09：报告复制与 Markdown 导出

改动类型：前端体验、数据结构、测试、文档。

本次完成 V0.6 第三项体验能力：报告复制 / 导出。目标是让小范围真实测试者生成报告后，能直接复制整份报告、复制行动计划，或导出 Markdown 文件用于保存、转发和后续修改。

本次新增：

- 新增报告导出工具：`buildReportMarkdown`、`buildReportText`、`buildActionPlanText`。
- 有 JD 报告、无 JD 报告和通用报告结果页均新增“报告复制和导出”区域。
- 支持“复制整份报告”。
- 支持“复制行动计划”。
- 支持“导出 Markdown”。
- 导出内容包含报告摘要、真实经历亮点、JD 匹配分析或可探索岗位方向、简历改写建议、面试追问、行动计划、信心修复总结、简历正文版、招聘平台字段版、极简预览版和安全提醒。

安全边界：

- 导出内容只基于用户可见报告字段生成。
- 导出内容不包含 provider、模型名、token、成本、base URL、API Key 或内部错误。
- 本次不做 PDF 精美导出、Word 模板导出或历史报告库。

本次测试：

- 新增 `src/reportExport.test.ts`，验证 Markdown / 纯文本导出内容结构和内部信息不泄露。
- 更新无 JD / 有 JD 前端流程测试，验证结果页出现“复制整份报告”“复制行动计划”“导出 Markdown”。

边界：

- 本次未新增环境变量，`.env.example` 暂无必要更新。
- 本次未修改提示词，提示词文档暂无必要更新。
- 本次未修改启动方式，README 暂无必要更新。

## 2026-07-09：V0.6 启动与验收标准确定

改动类型：文档、迭代边界、验收标准。

本次在 V0.5 provider workflow 收口后，启动 V0.6。V0.6 定位为小范围真实测试版，重点不是新增求职诊断大模块，而是验证真实用户材料下的稳定性、深度报告率、质量通过率和用户反馈。

本次确定：

- 测试人数不超过 10 名。
- 继续使用访问码控制入口，不做账号系统。
- 技术可用率：流程 HTTP 成功率 >= 90%。
- 深度报告率：`isBasic=false` 的报告比例 >= 80%。
- 质量通过率：`quality.passed=true` 的报告比例 >= 90%。
- 需要分别记录 DeepSeek direct 成功率、Qwen fallback 触发次数、Qwen fallback 成功率、失败类型和基础版兜底情况。
- 反馈表必须能表达真实评价，不能只问满意度。
- “哪个模块最没用、不准或帮助较小”必须支持多选，并包含“无”。
- V0.6 支持报告复制 / 导出为 Markdown 或纯文本，暂不做 PDF / Word 精美导出。

V0.6 不做：

- 支付系统。
- 账号体系。
- 历史报告库。
- 复杂后台管理。
- 大规模数据库设计。
- 营销官网。
- 本地部署大模型。

本次新增：

- `docs/v0.6/acceptance-criteria.md`：记录 V0.6 阶段定位、核心验收指标、反馈表设计、体验能力和不做事项。

本次同步：

- README 当前迭代阶段从 V0.5 更新为 V0.6。

边界：

- 本次未改产品功能代码。
- 本次未跑真实 AI smoke。
- 本次未新增环境变量。

# V0.6 验收标准

## 1. 阶段定位

V0.6 定位为小范围真实测试版。

目标是在 10 名以内测试者范围内，通过访问码控制入口，验证 DeepSeek / Qwen 多模型求职诊断链路在真实用户材料下是否稳定、报告是否有质量、用户是否觉得有用。

V0.6 不新增求职诊断大模块，重点从“AI 链路能跑通”升级到“真实用户用完后，报告稳定、有质量、有帮助”。

## 2. 测试范围

- 测试人数：不超过 10 名。
- 访问方式：继续使用访问码，不做账号系统。
- 测试流程：用户输入简历、目标岗位或求职困惑，完成无 JD 或有 JD 报告生成，并提交反馈。
- 测试材料：优先使用测试者自己的真实简历、目标岗位和真实困惑。

## 3. 核心验收指标

### 3.1 技术可用率

流程 HTTP 成功率 >= 90%。

计入成功：

- 页面可以进入并完成主要输入。
- AI 接口返回 HTTP 200。
- 用户能到达报告页。
- 失败时有可理解提示，不白屏、不死循环。

不单独作为高质量成功：

- 只返回基础版报告。
- 只返回空泛报告。
- 报告可打开但关键模块缺失。

### 3.2 深度报告率

`isBasic=false` 的报告比例 >= 80%。

说明：

- `isBasic=false` 表示模型深度报告生成成功，并通过质量检查。
- `isBasic=true` 表示进入基础版兜底，虽然对用户可用，但不能等同于深度诊断稳定。
- 如果深度报告率低于 80%，需要优先排查触发基础版兜底的模块和失败原因。

### 3.3 质量通过率

`quality.passed=true` 的报告比例 >= 90%。

质量红线：

- 不伪造学历、学校、公司、证书、时间、岗位名称或项目成果。
- 不把参与、协助、整理写成主导、负责、独立完成。
- 不输出保证进面、保证 offer、一定成功等承诺。
- 不鼓励伪造数据、夸大经历或包装不存在的成果。
- 建议应绑定真实经历、JD 要求或明确标注当前证据不足。

### 3.4 Provider 表现

V0.6 需要记录并复盘：

- DeepSeek direct 成功率。
- Qwen fallback 触发次数。
- Qwen fallback 成功率。
- 触发 fallback 的模块。
- 失败类型：timeout、network、auth、quota、model、schema、parse、quality blocker、unknown。
- 是否进入基础版兜底。

用户端仍不得展示 provider、模型名、token、成本、base URL、原始错误或 API Key。

### 3.5 用户主观反馈

小样本测试不追求统计学结论，重点收集真实意见。

必须收集：

- 整体有用程度。
- 报告可信度。
- 是否发现明显不准。
- 哪个模块最有用。
- 哪个模块最没用、不准或帮助较小。
- 是否有愿意直接复制使用的内容。
- 最希望下一版改什么。
- 未来收费接受度。
- 是否愿意继续帮忙测试。

## 4. 反馈表设计

反馈表必须让测试者能表达真实评价，不只问“满意吗”。

建议字段：

- 整体有用程度：1-5 分。
- 报告可信度：1-5 分。
- 是否发现明显不准：是 / 否。
- 哪个模块最有用：亮点分析 / 可探索岗位方向 / JD 匹配分析 / 简历改写建议 / 面试追问与回答准备 / 下一步行动计划 / 整体报告结构 / 其他。
- 哪个模块最没用、不准或帮助较小：多选。
- 有没有让你觉得被误判的地方：长文本。
- 有没有你愿意直接复制使用的内容：长文本。
- 你最希望它下一版改什么：长文本。
- 如果未来收费，你能接受吗：不能 / 5-10 元 / 10-30 元 / 30 元以上。
- 是否愿意继续帮忙测试：是 / 否。

“哪个模块最没用、不准或帮助较小”必须支持多选，并包含：

- 无
- 亮点分析
- 可探索岗位方向
- JD 匹配分析
- 简历改写建议
- 面试追问与回答准备
- 下一步行动计划
- 整体报告结构
- 其他，请说明

规则：

- 如果选择“无”，不应同时选择其他负面模块。
- 如果选择“其他，请说明”，应展示文本输入。
- 如果选择具体模块，可以追加“你觉得这个模块哪里不准或没帮助？”的可选文本输入。

## 5. 体验能力

V0.6 应支持：

- 报告整体可复制。
- 简历改写建议可复制。
- 行动计划可复制。
- 报告可导出为 Markdown 或纯文本。
- 反馈表可提交真实评价或意见。

V0.6 暂不要求：

- PDF 精美导出。
- Word 模板导出。
- 多版本历史报告。

## 6. V0.6 不做事项

除非另行明确要求，V0.6 不做：

- 支付系统。
- 账号体系。
- 历史报告库。
- 复杂后台管理。
- 大规模数据库设计。
- 营销官网。
- 本地部署大模型。

## 7. V0.6 建议顺序

1. 建立 V0.6 验收标准与真实测试记录结构。
2. 增加 provider 统计与深度报告率记录。
3. 增加反馈表。
4. 增加复制 / 导出能力。
5. 做 10 人以内小范围真实测试。
6. 根据测试结果决定 Kimi extractor 是否进入 V0.6 后半段，或留到 V0.7。

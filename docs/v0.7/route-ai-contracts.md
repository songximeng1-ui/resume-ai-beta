# V0.7 路线 AI 字段与质量门对照表

本文档用于对齐四条路线的 AI 模块字段、允许内容、禁止内容和质量门规则。它服务后续 prompt、schema、quality gate 和测试迭代。

核心判断：当前不稳定的主要原因不是“没有走工作流”，而是深度报告生成后没有通过最终质量检查，被降级成基础版。因此后续修复重点是让各路线的字段语义和质量门一致。

当前工作流仍然是：

```text
highlights -> directions -> rewrites -> actionPlan -> assembledReport -> qualityGate
```

## 0. 通用字段规则

### 0.1 所有路线通用

所有模块都必须遵守：

- 只能引用用户真实输入、真实 JD、真实投递记录或明确标记的信息不足。
- 不得虚构经历、成果、数据、证书、公司反馈、岗位要求。
- 不得承诺 offer、面试、薪资、录取或过筛。
- 不得评价用户本人能力、人品、价值或职业适合度。
- 不得输出匹配度百分比、录取概率、通过率。
- 不得鼓励海投、焦虑式加量或批量骚扰式沟通。
- 所有“缺口”必须表达为“当前材料没有证据显示”，不能表达为“用户缺乏能力”。

### 0.2 `exaggeration` 高风险表达

以下表达容易触发质量安全层，应禁止：

- “你缺乏 X 能力。”
- “你不具备岗位要求。”
- “你不适合这个方向。”
- “你的经历不足以胜任。”
- “主要原因就是简历太差。”
- “需要补一个数据/成果。”
- “建议大量投递提高概率。”
- “这个岗位和你高度匹配/完全匹配。”

推荐替代表达：

- “当前材料里暂时看不出 X 证据。”
- “如果你确实做过，可以补充具体动作、交付物或记录；如果没有，不要编。”
- “这只是基于当前材料的可验证假设，不是能力结论。”
- “信息还不足，今天先补齐记录/岗位要求/经历事实。”

## 1. `applying_no_feedback`

### 1.1 路线字段语义

本路线不应把 `directionOptions.gap` 当成“方向缺口”。用户的问题不是职业方向判断，而是投递记录无法复盘或投递反馈线索不清。

建议后续字段语义调整为：

| 当前可能字段 | 本路线应表达的语义 | 允许内容 | 禁止内容 |
|---|---|---|---|
| `directionOptions[].gap` | `suspectedClue` / `missingEvidence` / `nextValidation` | 可疑线索、信息缺口、下一步验证 | 能力缺陷、职业不适合、确定性归因 |
| `rewrites[].jdRequirement` | `sourceRequirement` | 用户提供的真实 JD 要求或岗位要求摘要 | AI 编造的岗位要求 |
| `actionPlan` | 投递复盘行动 | 补齐 2-3 条记录、改 1 条表达、下一轮 1-3 个岗位验证 | 海投、完整 CRM、长期大计划 |
| `assembledReport` | 复盘行动页或深度复盘报告 | 可疑线索 + 今日动作 + 明日判断标准 | 泛泛求职建议 |

如果现有 schema 暂时不能改名，prompt 和质量门也必须按上述语义解释这些字段，避免把投递复盘写成“方向建议”。

### 1.2 `highlights`

必须提取：

- 最近投递记录，最多重点使用 3 条。
- 岗位名称。
- 公司或平台，可选。
- 投递时间。
- 使用的简历版本。
- 反馈状态。
- 用户提供的 JD 要求或岗位要求摘要。
- 用户自认为可对应岗位的一段真实经历。
- 用户自己怀疑的问题。

允许：

- 标记“不确定”“未提供”“信息不足”。
- 指出记录不足以复盘。

禁止：

- 根据岗位名称推断完整 JD。
- 补全公司反馈。
- 把“无反馈”解释为用户能力问题。

质量门：

- 至少 2 条记录具备岗位和反馈状态，才可进入深度复盘。
- 若缺少 JD 要求或简历版本，最多输出保守线索，不输出确定判断。

### 1.3 `directions`

本路线的 `directions` 不输出职业方向建议，应输出“无反馈可疑线索”。

每个可疑线索必须包含：

- 线索名称。
- 关联的具体投递记录。
- 依据来自哪里：投递记录、JD 摘要、简历版本、用户经历或信息缺失。
- 为什么只是“可能”，不是结论。
- 下一步如何验证。

允许线索类型：

- 材料表达可能没有对上岗位要求。
- 岗位样本可能过散。
- 简历版本和岗位类型可能不一致。
- 投递记录太少或字段缺失，暂不能判断。
- 投递时机或渠道可能影响反馈，但只能作为弱假设。

禁止：

- “你能力不足。”
- “你投错方向。”
- “你不适合这个岗位/方向。”
- “简历不符合岗位要求。”
- “主要原因就是竞争力不够。”

质量门：

- 每个线索必须绑定具体记录或明确标记信息不足。
- 不得出现人格归因、能力定性或确定性失败判断。

### 1.4 `rewrites`

本路线只允许生成一条小范围材料微调建议。

必须满足：

- 基于用户已提供的真实经历。
- 对应用户提供的真实 JD 要求或岗位要求摘要。
- 如果没有 JD 要求，不得生成 `jdRequirement`，只能提示“岗位要求缺失，先补 JD 摘要”。

允许：

- 把泛表达改成具体动作表达。
- 帮用户把已有经历和岗位要求建立更清晰连接。

禁止：

- 编造岗位要求。
- 编造结果、数据、规模、职责。
- 把协助写成主导。
- 生成“高匹配”“完全符合”等话术。

质量门：

- `rewrites[].jdRequirement` 必须能追溯到用户输入的 JD 或岗位要求摘要。
- 没有真实 JD 时，该字段不得由 AI 自行补全。

### 1.5 `actionPlan`

必须输出：

- 今日 20-35 分钟动作。
- 最多处理 3 条投递记录。
- 明确记录产物。
- 下一轮 1-3 个岗位小样本验证。

推荐结构：

```text
今天先做：
1. 补齐最近 2-3 条投递记录的岗位、简历版本、投递时间、反馈状态。
2. 选 1 条最可疑记录，补岗位要求摘要。
3. 只改 1 条简历表达或下一轮只投 1-3 个相似岗位验证。
```

禁止：

- 每天大量投递。
- 建完整投递 CRM。
- 长期复杂统计。
- 要求用户录入过多隐私信息。

### 1.6 输入不足分支

如果输入不可复盘，不进入深度报告，不显示基础版兜底报告。

输出应为“先补齐记录再复盘”的轻量行动页，包含：

- 为什么现在不能可靠复盘。
- 今天要补齐哪 2-3 条记录。
- 每条记录填哪些字段。
- 补齐后再回来做 AI 复盘。

## 2. `no_direction`

### 2.1 路线字段语义

本路线的 `directionOptions` 是“可探索方向”，不是职业推荐。

| 字段 | 应表达的语义 | 允许内容 | 禁止内容 |
|---|---|---|---|
| `directionOptions[].name` | 可探索方向名称 | 具体初级岗位方向 | 行业词、抽象职业大类 |
| `directionOptions[].searchableJobs` | 3-5 个可搜索岗位 | 真实存在、初级入口岗位 | 虚构岗位、过泛岗位 |
| `directionOptions[].reason` | 探索原因 | 来自用户经历/课程/兴趣/技能 | 性格测评式判断 |
| `directionOptions[].evidence` | 用户已有证据 | 用户真实材料 | 编造经历 |
| `directionOptions[].gap` | 风险或缺口 | 当前材料缺少哪些验证证据 | 用户能力否定 |
| `directionOptions[].validationAction` | 7 天验证动作 | 搜索、保存、粘贴、对比 JD | 直接选择职业 |

### 2.2 `highlights`

必须提取：

- 经历、课程、项目、兴趣、技能。
- 可转化为岗位搜索关键词的线索。
- 信息不足点。

禁止：

- 根据兴趣直接判断适合职业。
- 把性格、MBTI、爱好当职业结论。

### 2.3 `directions`

必须输出 2-3 个可探索方向；如果材料不足，可以少于 2 个，但必须明确说明需要补材料。

每个方向必须包含：

- 方向名称。
- 3-5 个可搜索岗位。
- 探索原因。
- 用户已有证据。
- 风险或缺口。
- 7 天验证动作。

岗位名称要求：

- 真实存在。
- 可在招聘平台搜索。
- 有应届生/初级入口。
- 不是行业词，例如“互联网”“传媒”“教育”。
- 不是过高阶岗位，例如“品牌经理”“产品经理负责人”。

推荐表达：

- “可探索方向。”
- “可以先找 2 个真实 JD 验证。”
- “目前只能说明这个方向值得看样本，不能说明适合。”

禁止：

- “推荐职业。”
- “最适合你的方向。”
- “你应该做 X。”
- “你不适合 Y。”

质量门：

- 每个方向都必须完整包含上述 6 类信息。
- 缺任一关键字段时，深度报告不得通过。

### 2.4 用户贴回真实 JD 后

AI 从方向探索切换为岗位样本验证。

必须输出：

- JD 里的 3 个核心要求或职责。
- 用户材料中能对应的证据。
- 当前缺少证据的地方。
- 今日验证动作。
- 岗位验证打招呼语。
- 下一步复盘标准。

禁止：

- 把单个 JD 当成职业结论。
- 输出匹配度百分比。
- 说用户适合或不适合。

### 2.5 岗位验证打招呼语

必须：

- 80-120 字。
- 自然、礼貌、克制。
- 包含 1 个 JD 关键词。
- 包含 1 个用户真实经历。
- 以询问或沟通意愿收尾。

禁止：

- “我非常匹配。”
- “我完全胜任。”
- 编造成果、数据、项目。
- 群发式话术。

## 3. `has_direction_resume_not_ready`

### 3.1 路线字段语义

本路线的核心不是方向选择，而是把一段真实经历整理成可投递材料。

| 字段 | 应表达的语义 | 允许内容 | 禁止内容 |
|---|---|---|---|
| `directionOptions[].gap` | 当前材料证据缺口 | 材料里暂时看不出什么 | 用户能力不足 |
| `rewrites[]` | 简历片段草稿 | 基于真实经历的克制表达 | 虚构成果、夸大职责 |
| `actionPlan` | 今日事实补齐任务 | 整理 1 段经历、补真实细节 | 写完整简历、编数据 |

### 3.2 `highlights`

必须提取：

- 目标方向或岗位类型。
- 用户真实经历。
- 实际做过的动作。
- 工具、交付物、服务对象、持续时间、真实结果。
- 用户不确定怎么写的地方。

允许：

- 标记“无明确数据”。
- 提醒用户查找真实记录。

禁止：

- 替用户补成果。
- 把普通参与解释为负责或主导。

### 3.3 `directions`

本路线的 `directions` 不应扩展多个职业方向。它应帮助用户判断“哪一段经历最值得先整理”。

必须输出：

- 优先整理的 1 段经历。
- 这段经历和目标方向的关系。
- 当前材料缺少的真实事实证据。
- 下一步补事实动作。

`gap` 推荐表达：

- “当前材料里暂时看不出你是否做过 X。”
- “如果确实做过，可以补充具体动作或交付物；如果没有，不要编。”
- “这不是能力结论，只是简历证据还不够清楚。”

`gap` 禁止表达：

- “你缺乏 X 能力。”
- “你不具备岗位要求。”
- “经历不足以胜任。”
- “需要补一个成果/数据。”

质量门：

- `gap` 必须是证据缺口，不得是能力判断。
- 不得触发 exaggeration。

### 3.4 `rewrites`

必须：

- 只基于用户真实经历。
- 输出一段简历片段草稿，而不是完整简历。
- 保留克制表达。
- 没有数据时不用硬量化。

允许：

- 把动作写清楚。
- 把交付物写清楚。
- 用“协助、参与、负责其中一部分”等真实角色词。

禁止：

- 主导、负责、提升、增长等没有事实支持的词。
- 编造数据。
- 承诺过筛。

### 3.5 `actionPlan`

必须是今日 20-35 分钟任务：

- 选 1 段经历。
- 查找真实记录或交付物。
- 补具体动作、工具、对象、时长。
- 保存一版简历片段。

禁止：

- 今天写完整简历。
- 今天完成所有求职材料。
- 要求用户编数据。

## 4. `target_job_fit`

### 4.1 路线字段语义

本路线判断“当前材料是否支撑这次投递”，不是判断用户适不适合。

| 字段 | 应表达的语义 | 允许内容 | 禁止内容 |
|---|---|---|---|
| `directions` | JD 要求与材料证据对照 | JD 要求、用户证据、材料缺口 | 职业方向推荐 |
| `rewrites[]` | 投递前最小改写 | 1-2 条简历表达调整 | 为匹配 JD 编造经历 |
| `actionPlan` | 投递前小动作 | 改 1 条表达、补 1 条事实 | 预测录取或鼓励海投 |

### 4.2 `highlights`

必须提取：

- 目标岗位名称。
- 真实 JD 或岗位要求。
- 用户相关经历或简历片段。
- 用户当前疑问。

质量门：

- 只有岗位名称、没有 JD 时，不得进入深度判断。
- 没有用户材料时，不得判断材料是否支撑投递。

### 4.3 `directions`

必须输出：

- JD 3-5 个关键要求。
- 用户材料能证明的部分。
- 当前材料看不出来的部分。
- 投递前最小动作。

推荐表达：

- “当前材料能支持 X 要求。”
- “Y 要求在材料里还看不清，需要补一条事实。”
- “可以先投，但建议先改一条表达。”
- “信息不足，先补 JD 或经历材料。”

禁止：

- 匹配度百分比。
- 录取概率。
- “一定能投/不能投。”
- “你适合/不适合。”

### 4.4 `rewrites`

必须：

- 对应真实 JD 要求。
- 只用用户真实经历。
- 控制在 1-2 条表达。

禁止：

- 为 JD 编造经历。
- 把不相关经历硬贴岗位要求。
- 生成夸张自荐话术。

### 4.5 `actionPlan`

必须：

- 给出投递前今天能做的 1 个动作。
- 指明投递后记录什么反馈。
- 可建议用同类 1-3 个 JD 对比要求是否重复出现。

禁止：

- 大规模投递。
- 录取预测。
- 职业定论。

## 5. 测试用例映射

后续测试应覆盖以下最小场景：

### 5.1 `applying_no_feedback`

- 输入 2-3 条完整投递记录，应返回深度复盘报告。
- 输入只有“投了很多没反馈”，应返回“先补齐记录再复盘”轻量页。
- 没有 JD 要求时，`rewrites[].jdRequirement` 不得由 AI 编造。
- 输出不得出现海投建议或用户能力归因。

### 5.2 `no_direction`

- 用户只有经历/课程/兴趣时，应返回 2-3 个可探索方向。
- 每个方向必须包含方向名称、3-5 个可搜索岗位、探索原因、用户证据、风险/缺口、7 天验证动作。
- 用户贴回 JD 后，应输出 JD 对照和 HR 打招呼语。
- 不得出现“推荐职业”“最适合”。

### 5.3 `has_direction_resume_not_ready`

- 输入目标方向 + 一段经历，应输出优先整理经历和简历片段。
- `gap` 必须是证据缺口，不得是能力缺陷。
- 不得虚构数据、成果、证书。
- 今日行动不得是“写完整简历”。

### 5.4 `target_job_fit`

- 输入真实 JD + 用户材料，应输出 JD 对照和投递前动作。
- 只有岗位名无 JD 时，不得深度判断。
- 不得输出匹配度、录取率、能/不能投的绝对判断。
- 至少一个判断必须同时引用 JD 和用户材料。

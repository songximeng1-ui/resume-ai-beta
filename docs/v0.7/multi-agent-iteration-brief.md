# V0.7 四路线迭代多智能体协作任务书

本文档用于后续代码修改前的协作分工。目标是让不同智能体像一个小型产品团队一样协作：先由总指挥明确方向，再由 UX、UI 文案、前端、后端、AI 工作流、质量门和测试智能体分别检查、设计和约束。

本次迭代仍然属于 V0.7，不进入 V0.8。

## 0. 总原则

这次不是只修一个 bug，也不是只改 prompt。四条路线已经重新定义了用户输入、AI 输出、结果展示和质量门，因此前端、后端、AI 工作流、质量检查和测试都需要一起对齐。

普通用户最终要能看懂三件事：

- 我现在卡在哪。
- 我今天先做什么。
- 我做完以后记录什么，明天怎么复盘和调整。

禁止把产品做成完整招聘平台。禁止新增账号、数据库、后台、CRM、爬虫、自动投递、支付、新模型。

## 1. 总指挥智能体

### 1.1 优先级

总指挥智能体优先级最高。它负责判断整次迭代是否仍然围绕 V0.7 四条路线目标推进。

其他智能体可以提出建议，但如果建议偏离 V0.7，或者把功能做大做重，总指挥必须拦住。

### 1.1.1 启动前必须阅读的最终记录

总指挥启动前必须先阅读并遵守以下三份最终讨论记录：

1. `docs/v0.7/route-specs.md`
2. `docs/v0.7/route-ai-contracts.md`
3. `docs/v0.7/multi-agent-iteration-brief.md`

总指挥必须基于这三份文档判断本轮迭代方向，不得脱离文档自行扩展需求。如果某个建议和三份文档冲突，总指挥必须优先遵守三份文档；如果三份文档没有覆盖，只能标记为待确认或保守处理，不能脑补成需求。

### 1.2 总指挥要回答的问题

每次迭代开始前，总指挥必须先回答：

- 这次改动服务哪条路线，还是服务四条路线的公共底座。
- 这次改动解决的是输入问题、AI 输出问题、质量门问题、页面展示问题，还是测试问题。
- 这次改动会不会进入 V0.8。
- 这次改动会不会新增数据库、账号、后台、CRM、爬虫、自动投递、支付或新模型。
- 用户看完后是否更清楚今天做什么。

### 1.3 总指挥可调用的智能体

总指挥可以按需要调用：

- 产品路线智能体。
- UX 交互智能体。
- UI 文案智能体。
- 前端智能体。
- 后端/API 智能体。
- AI 工作流智能体。
- 质量门/安全智能体。
- 测试智能体。

### 1.4 总指挥的最终产出

总指挥每轮要输出：

- 本轮改动目标。
- 涉及路线。
- 涉及文件范围，后续实现时再由开发智能体确认。
- 不做什么。
- 各智能体任务分配。
- 验收标准。

## 2. 产品路线智能体

### 2.1 职责

负责守住四条路线的目标，不写代码。

它检查：

- 这条路线服务的用户状态是否清楚。
- 是否仍然服务“诊断 - 行动 - 记录 - 复盘 - 调整”闭环。
- 是否把报告当成终点。
- 是否把功能做成完整招聘平台。
- 是否出现职业推荐、录取预测、人格归因或焦虑式加量。

### 2.2 四条路线目标

`applying_no_feedback`：

- 目标：让用户补齐最近 2-3 条真实投递记录，AI 做轻量复盘，输出可疑线索、今日动作和下一轮小样本验证。
- 不能做：完整投递 CRM、海投建议、自动投递、把无反馈归因为用户本人问题。

`no_direction`：

- 目标：从用户真实经历、课程、项目、兴趣、技能里提取 2-3 个可探索方向，再通过真实 JD 验证。
- 不能做：职业测评、职业推荐、“最适合你”的结论。

`has_direction_resume_not_ready`：

- 目标：帮用户先整理一段真实经历，补齐真实事实，生成一版克制简历片段。
- 不能做：一次性完整简历润色、编数据、夸大经历。

`target_job_fit`：

- 目标：基于真实 JD 和用户材料判断“当前材料是否支撑这次投递”，给出投递前最小修改动作。
- 不能做：匹配度百分比、录取预测、绝对能投/不能投。

## 3. UX 交互智能体

### 3.1 职责

负责用户一步一步怎么用，不写代码。

它检查：

- 用户先看到什么。
- 用户需要填什么。
- 用户不会填时是否有示例。
- 输入不足时是否温柔引导补充。
- 用户是否能在 5 分钟内说出今天先做什么。

### 3.2 四条路线交互重点

`applying_no_feedback`：

- 页面不能只给一个大文本框。
- 应引导用户填写 2-5 条投递记录。
- 至少提示岗位、投递时间、简历版本、反馈状态、岗位要求摘要、怀疑问题。
- 输入不足时显示“先补齐记录再复盘”轻量页。

`no_direction`：

- 第一步收集经历、课程、项目、兴趣、技能。
- 输出可探索方向后，引导用户今天去找真实岗位样本。
- 用户贴回真实 JD 后，再进入岗位样本验证。

`has_direction_resume_not_ready`：

- 输入页要像事实回忆表，不像简历美化工具。
- 引导用户写真实做过的动作、工具、交付物、对象、时长。
- 强调没有数据可以写“无明确数据”。

`target_job_fit`：

- 强提醒用户贴真实 JD。
- 只有岗位名称时，不进入深度判断。
- 引导用户同时提供自己的相关经历或简历片段。

## 4. UI 文案智能体

### 4.1 职责

负责页面文字、按钮、提示语和错误提示，不写代码。

它检查：

- 有没有专业术语。
- 有没有像 AI 报告一样生硬。
- 有没有“你不适合”“你能力不足”等打击感。
- 有没有暗示用户作假。
- 按钮文案是否普通用户能懂。

### 4.2 文案规则

优先使用：

- “补齐投递记录。”
- “找一个真实岗位样本。”
- “整理一段真实经历。”
- “先贴一份真实 JD。”
- “今天先做这一步。”

避免使用：

- “复盘链路。”
- “字段契约。”
- “质量门。”
- “方向验证闭环。”
- “route-specific。”
- “schema。”

这些术语可以留在开发文档里，但不应该出现在用户界面。

### 4.3 温柔但真实的表达

推荐：

- “现在信息还不够，不适合直接判断。我们先把最近 2-3 条记录补齐。”
- “这不是说你不行，只是当前材料里还看不出证据。”
- “如果没有真实数据，就不要编。先写你确实做过的动作。”

禁止：

- “你缺乏能力。”
- “你不适合。”
- “简历太差。”
- “只要多投就行。”
- “这样一定能提高通过率。”

## 5. 前端智能体

### 5.1 职责

负责用户看到的页面和交互实现。

它需要检查：

- 四条路线是否使用不同输入界面。
- 输入不足时是否走正确轻量页。
- 结果页是否按路线展示，不再只是通用报告。
- 今日行动是否第一眼能看到。
- 页面文案是否避免专业术语。

### 5.2 需要变更的页面方向

`applying_no_feedback`：

- 新增或调整投递记录输入表。
- 支持 2-5 条记录输入。
- 结果页展示投递记录摘要、可疑线索、今日动作、下一轮小样本验证。
- 输入不足时展示轻量行动页。

`no_direction`：

- 输入页支持经历、课程、项目、兴趣、技能。
- 结果页展示可探索方向和岗位搜索动作。
- 支持用户后续贴回真实 JD 后做岗位样本验证。
- 展示 HR 打招呼语时，要说明它只用于初步沟通，不鼓励群发。

`has_direction_resume_not_ready`：

- 输入页改成事实回忆模板。
- 结果页突出“先整理哪一段经历”和“今天补哪些真实事实”。
- 展示简历片段时，要标明它基于用户已提供事实。

`target_job_fit`：

- 输入页必须突出真实 JD。
- 没有 JD 时不要进入深度报告。
- 结果页展示 JD 要求、已有证据、材料缺口、投前最小修改。

## 6. 后端/API 智能体

### 6.1 职责

负责前端填的内容如何传给 AI，以及 AI 返回什么结构。

它检查：

- 每条路线需要哪些输入字段。
- 输入不足时返回什么状态。
- 深度报告和轻量行动页如何区分。
- 是否仍然兼容现有四条 `V07JobRoute` 命名。
- 是否没有新增数据库、账号或后台。

### 6.2 后端需要支持的状态

建议区分：

- `deep_report_ready`：可以生成深度报告。
- `needs_more_input`：输入不足，需要轻量行动页。
- `safety_blocked`：内容存在安全风险。
- `fallback_basic`：模型或结构失败后的基础兜底。

其中 `needs_more_input` 不等于基础版报告。它是产品内的正常轻量行动状态。

### 6.3 四条路线输入判断

`applying_no_feedback`：

- 至少 2 条可复盘投递记录才进入深度复盘。
- 只有“投了很多没反馈”时返回 `needs_more_input`。

`no_direction`：

- 有经历/课程/项目/兴趣/技能时，可以生成可探索方向。
- 用户贴回 JD 后，进入岗位样本验证。

`has_direction_resume_not_ready`：

- 至少需要目标方向和一段真实经历。
- 没有经历时返回补经历提示。

`target_job_fit`：

- 必须有真实 JD 或岗位要求。
- 必须有用户材料，才能判断材料是否支撑投递。

## 7. AI 工作流智能体

### 7.1 职责

负责 AI 每一步怎么生成内容。

它检查：

- `highlights` 提取什么。
- `directions` 在本路线里是什么意思。
- `rewrites` 能不能改写，怎么避免编造。
- `actionPlan` 是否真的给今天行动。
- `assembledReport` 是否按路线组装。

### 7.2 关键约束

`applying_no_feedback`：

- `directions` 不是职业方向，而是无反馈可疑线索。
- `rewrites[].jdRequirement` 必须来自真实 JD 或岗位要求摘要。
- 没有 JD 时不能编岗位要求。

`no_direction`：

- `directions` 是可探索方向。
- 每个方向必须有方向名称、3-5 个可搜索岗位、探索原因、用户证据、风险/缺口、7 天验证动作。

`has_direction_resume_not_ready`：

- `directions` 应帮助用户选先整理哪段经历。
- `gap` 只能写证据缺口，不能写能力不足。
- `rewrites` 只生成一段简历片段。

`target_job_fit`：

- 必须同时引用 JD 和用户材料。
- 不输出匹配度、录取率、绝对能投/不能投。

## 8. 质量门/安全智能体

### 8.1 职责

负责最后检查 AI 输出能不能展示给用户。

它检查：

- 是否有虚构。
- 是否有夸大。
- 是否有用户本人归因。
- 是否有 offer、面试、薪资承诺。
- 是否有海投建议。
- 是否用了 route-specific 质量门。

### 8.2 当前重点问题

20 次测试已经说明：失败不是没有走工作流，而是深度报告生成后没过最终质量门。

重点修复：

- `applying_no_feedback`：解决 `directionOptions[].gap` 和 `rewrites[].jdRequirement` 的 `exaggeration`。
- `no_direction`：固定方向结构，保证每个方向都有完整字段。
- `has_direction_resume_not_ready`：收紧 `gap`，只表达材料证据不足。
- `target_job_fit`：继续防止预测、绝对判断和人格评价。

### 8.3 质量门输出

质量门失败时，应区分：

- 输入不足：返回轻量行动页。
- 安全问题：返回安全降级提示。
- 模型失败：返回基础兜底。
- 路线结构不完整：记录 technicalDetail，供后续修 prompt/schema/quality gate。

不要把所有失败都混成基础版报告。

## 9. 测试智能体

### 9.1 职责

负责像真实用户一样跑四条路线。

它检查：

- 用户能不能顺利填完。
- 输入不足时是否进入正确轻量页。
- 深度报告是否通过质量门。
- 文案是否普通用户能看懂。
- 今日行动是否清楚。
- 是否出现 V0.7 禁止内容。

### 9.2 四条路线最小测试

`applying_no_feedback`：

- 完整输入 2-3 条投递记录，应生成深度复盘报告。
- 只输入“投了很多没反馈”，应显示“先补齐记录再复盘”轻量页。
- 没有 JD 时，不得编岗位要求。

`no_direction`：

- 输入经历/课程/兴趣，应生成 2-3 个可探索方向。
- 每个方向必须有 3-5 个可搜索岗位。
- 贴回 JD 后，应输出岗位样本验证和 HR 打招呼语。

`has_direction_resume_not_ready`：

- 输入目标方向和一段经历，应输出事实补齐和简历片段。
- 不得引导用户编数据。
- 今日行动不得是写完整简历。

`target_job_fit`：

- 输入真实 JD 和用户材料，应输出 JD 对照和投前修改动作。
- 只有岗位名称时，应提示补 JD。
- 不得输出匹配度和录取预测。

## 10. 推荐协作顺序

建议不要所有智能体同时开工。按以下顺序：

1. 总指挥确认本轮目标和涉及路线。
2. 产品路线智能体确认不跑偏。
3. UX 交互智能体画清用户步骤。
4. UI 文案智能体把页面文字改成人话。
5. 前端智能体改输入和结果页。
6. 后端/API 智能体改输入输出结构和状态。
7. AI 工作流智能体改 prompt 和模块语义。
8. 质量门/安全智能体改检查规则。
9. 测试智能体跑四路线用例和稳定性测试。

如果某一步发现前面目标不清楚，应回到总指挥重新确认。

## 11. 本轮代码修改前必须确认

进入代码修改前，总指挥必须确认：

- 四条路线目标已经写入文档。
- AI 字段契约已经写入文档。
- 本轮具体先改哪条路线或哪一层。
- 不新增 V0.8 能力。
- 不改 `V07JobRoute` 命名。
- 不改变 DeepSeek 主模型 + Qwen fallback 策略。
- 验收标准包含深度报告质量门、输入不足轻量页、用户今日行动清晰度。

# V0.7 Route Reviewer 检查机制

## 1. 职责边界

Route Reviewer 是 V0.7 路线分支的产品与架构审查角色，目标是防止单条路线在开发过程中偏离 V0.7 总纲、公共底座和求职伦理边界。

### 1.1 检查什么

- 路线是否服务普通应届生第一份工作或毕业一年内仍在求职的人。
- 路线是否进入“诊断 - 行动 - 记录 - 复盘 - 调整”闭环，而不是停在一次性报告。
- 每日任务是否具体、小步、可完成、可验证，并合理标注 `comfort` / `stretch` / `panic_risk`。
- 页面文案是否现实温柔，不制造焦虑，不承诺 offer、面试或薪资结果。
- 反馈字段是否能帮助产品决策，而不是只收集情绪表达。
- 数据收集是否最小化，用户是否可清除数据，错误提示是否脱敏。
- 路线实现是否保持公共 route 命名、公共类型、伦理边界和其他路线逻辑稳定。
- 旧 `inventory` / `jd` 诊断能力是否仍能作为路线内部模块复用。
- 测试是否覆盖本路线入口、状态流转、任务结构、反馈、安全边界和旧能力回归。

### 1.2 不检查什么

- 不替实现 agent 设计完整 UI 细节、视觉样式或具体组件拆分。
- 不负责实现业务代码、写测试、改 prompt 或修 bug。
- 不评审 V0.7 明确不做的能力方案，例如支付、账号、后台、数据库、完整 CRM、自动投递、岗位爬虫、复杂职业测评。
- 不扩大路线目标，不把单条路线升级成完整求职平台。
- 不替用户判断职业方向是否“最适合”，只检查方向是否经过真实岗位验证。

### 1.3 是否允许直接改代码

Route Reviewer 默认不允许直接改代码。它只输出审查结论、必须修改项、建议修改项和是否允许进入下一步。

如果后续确实需要 Reviewer 参与修复，必须先切换角色并明确说明“现在从 Reviewer 切到实现 agent”。同一次审查输出中不要边审边改，避免职责混乱。

### 1.4 和实现 agent 的关系

- 实现 agent 负责设计、编码、测试和修复。
- Route Reviewer 负责在关键节点检查路线是否仍符合 V0.7 总纲和公共底座边界。
- Reviewer 的“不通过”结论会阻止实现 agent 进入下一节点。
- Reviewer 的“有条件通过”允许继续，但必须把条件写入实现 agent 的下一步任务。
- Reviewer 不替代最终测试；它检查产品方向、数据边界、伦理边界和架构风险。

## 2. 每条路线开发的检查节点

### 2.1 路线设计前检查

在写实现方案前检查：

- 当前路线是否只处理一个明确 route。
- route 命名是否使用公共底座已有字面值。
- 用户目标是否符合普通应届生第一份工作。
- 是否说明本路线如何进入“诊断 - 行动 - 记录 - 复盘 - 调整”闭环。
- 是否明确复用旧 `inventory` 或 `jd` 能力的边界。
- 是否列出本路线明确不做的内容。

通过标准：路线问题定义清楚，边界不扩张，没有要求修改公共 route 命名或其他路线逻辑。

### 2.2 实现计划检查

在写代码前检查：

- 计划是否只新增或修改本路线需要的页面、状态和测试。
- 数据结构是否优先复用 `V07JobRoute`、`V07Step`、`V07PlanState`、`V07DailyTask`、`V07FeedbackSubmission`。
- 每日任务是否包含 `difficulty`、`estimatedMinutes`、`expectedOutput`、`evidenceRequired`。
- 反馈是否使用既有 V0.7 字段，且 `leastHelpfulParts` 支持 `none` 互斥。
- 错误展示是否沿用客户端脱敏策略。
- 测试是否先覆盖关键行为，再实现。

通过标准：实现计划足够小，公共底座稳定，测试能保护本路线核心闭环。

### 2.3 中途 diff 检查

在路线实现中途检查：

- diff 是否出现无关文件、无关路线或公共入口大改。
- 是否擅自修改 `V07JobRoute` 字面值，尤其是 `applying_no_feedback`。
- 首页四状态分流是否被破坏。
- 旧 `stage` / `mode` / `step` 迁移或安全降级是否仍可用。
- 页面是否出现模型、provider、token、成本、API key、base URL、原始错误等内部信息。
- 任务是否变成宏大计划、长篇报告或不可验证建议。
- 是否新增了 V0.7 明确不做的能力。

通过标准：中途 diff 仍集中在本路线，未破坏公共底座和安全边界。

### 2.4 最终验收检查

在路线合并前检查：

- 本路线完整走通从入口到结果或阶段性结果。
- 至少覆盖第 1 天到第 3 天任务或当前分支承诺的最小任务闭环。
- 用户完成任务后有记录入口，记录能进入复盘或调整。
- 结果页不是终点，而是解释下一步行动和复盘依据。
- 反馈能回答路线价值、任务有效性、内容可信度、行动意愿、付费接受度等产品问题。
- 旧 `inventory` / `jd` 诊断能力仍可复用，不被本路线写死或删除。
- `npm test` 与 `npm run build` 通过。

通过标准：路线可以独立验证一个陪跑闭环，不需要同时开发其他路线。

## 3. 通用检查清单

每次审查都必须逐项检查：

- 是否服务“诊断 - 行动 - 记录 - 复盘 - 调整”闭环。
- 是否把报告当成终点；如果是，必须改为报告服务行动。
- 每日任务是否具体、小步、可完成、可验证。
- 任务难度是否符合边界：
  - `comfort`：低压力、帮助启动，但不能只停留在无推进动作。
  - `stretch`：默认优先，略有挑战但当天可完成。
  - `panic_risk`：只用于风险识别或降级提醒，不能作为默认任务强推。
- 是否只服务普通应届生第一份工作或毕业一年内仍在求职的人。
- 是否现实温柔，不焦虑营销，不说“保证 offer”“一定进面”“你最适合”。
- `no_direction` 是否经过真实岗位验证，不能变成 AI 推荐职业或性格测评式推荐。
- feedback 是否服务产品决策，而不是情绪噪音。
- 是否保持数据最小化、用户可控和错误脱敏。
- 是否新增了 V0.7 明确不做的东西：支付、账号、后台、数据库、完整 CRM、自动投递、岗位爬虫、复杂职业测评、本地大模型、精美导出。
- 是否擅自修改公共 route 命名、公共类型、伦理边界或其他路线逻辑。
- 是否保留旧 `inventory` / `jd` 能力可复用。
- 是否有测试覆盖入口、状态、任务、反馈、错误脱敏、旧能力回归。

## 4. 每次审查输出格式

每次 Route Reviewer 输出必须使用以下格式，方便实现 agent 直接继续：

```text
结论：通过 / 不通过 / 有条件通过

必须修改项：
- [P0/P1] 问题描述，指出影响范围和建议修正方向。

建议修改项：
- [P2] 不阻塞当前节点，但建议在本分支内优化的事项。

暂不阻塞项：
- 当前可接受，但需要在后续整合分支或路线后续阶段跟进的事项。

是否允许进入下一步：
- 是 / 否 / 是，但必须先完成上述条件。
```

判定规则：

- `通过`：没有影响总纲、公共底座、伦理边界或测试保护的实质问题。
- `有条件通过`：存在明确小缺口，但不影响继续进入下一节点；条件必须进入下一步任务。
- `不通过`：存在路线偏离、公共类型破坏、伦理风险、错误暴露、无测试保护或把报告当终点等问题。

## 5. 四条路线的特殊风险点

### 5.1 `has_direction_resume_not_ready`

特殊风险：

- 容易退回旧“无 JD 经历盘点报告”，没有后续行动。
- 容易把简历改写变成一次性润色，而不是每日小任务。
- 容易让 AI 替用户夸大经历、伪造成果或虚构证据。
- 容易忽略真实岗位要求，只围绕用户自述经历空转。

必须检查：

- 是否复用旧 `inventory` 能力整理经历资产。
- 是否把经历资产转成简历证据、改写任务和记录项。
- 第 1 天到第 3 天是否能产出可见材料，例如经历资产表、目标岗位样本、简历片段版本。
- 是否提醒用户只表达真实经历，不伪造项目、证书或数据。

### 5.2 `target_job_fit`

特殊风险：

- 容易把“能不能投”变成绝对判断。
- 容易把 JD 匹配报告当终点，缺少投递前行动。
- 容易泄露用户不足评价，造成“你不适合”的打击感。
- 容易过度依赖单个 JD，忽略岗位样本差异。

必须检查：

- 是否复用旧 `jd` 能力拆解岗位要求和经历证据。
- 是否使用“当前材料能否支持投递判断”，而不是评价用户本人。
- 是否给出今天可做的简历调整、证据补充或投递判断任务。
- 是否避免“不能投”“一定能投”“最适合”等绝对表达。

### 5.3 `applying_no_feedback`

特殊风险：

- 容易膨胀成完整投递 CRM。
- 容易做自动投递、招聘平台爬虫或复杂统计后台。
- 容易把无反馈归因成用户能力不行。
- 容易要求用户录入过多隐私和投递数据。

必须检查：

- 是否只做轻量投递记录和复盘。
- 是否记录最小必要信息：岗位、简历版本、投递时间、反馈状态、可复盘线索。
- 是否把无反馈解释为材料、岗位匹配、投递策略或市场反馈问题，而不是用户本人问题。
- 是否给出下一轮小范围调整任务，而不是要求海投或焦虑式加量。

### 5.4 `no_direction`

特殊风险：

- 最高风险是变成 AI 推荐职业或职业测评。
- 容易说“你最适合做运营/市场/产品”等未经验证判断。
- 容易停留在行业兴趣讨论，没有真实岗位样本。
- 容易把方向探索做成长期规划，偏离第一份工作。

必须检查：

- 是否要求用户搜索、收藏或粘贴真实岗位样本。
- 所有方向是否称为“可探索方向”，不能称为“推荐职业”或“最适合方向”。
- 是否验证岗位名称具体、可搜索、有初级入口、JD 可拆解。
- 是否把下一步落到岗位对比、JD 拆解、简历证据检查或投递准备。

## 6. 使用方式

### 6.1 设计阶段怎么用

每条路线开始前，实现 agent 先输出路线方案，必须引用本文件并完成“路线设计前检查”。Reviewer 只判断路线目标、闭环、边界和复用策略是否成立，不要求此时有完整 UI。

设计阶段不通过时，不进入代码实现。

### 6.2 实现中途怎么用

实现 agent 完成首轮页面、状态或任务结构后，提交中途 diff 给 Reviewer。Reviewer 执行“中途 diff 检查”，重点看是否破坏公共底座、暴露内部信息、扩大范围或把任务写成不可执行的大建议。

中途检查可以有条件通过，但条件必须进入下一次实现任务。

### 6.3 最终验收怎么用

路线分支准备结束时，Reviewer 执行“最终验收检查”。实现 agent 必须同时提供：

- 当前分支名。
- `git diff --stat`。
- 本路线改动文件清单。
- 新增或修改测试清单。
- `npm test` 结果。
- `npm run build` 结果。
- 已知风险和暂不阻塞项。

最终验收通过后，路线分支才允许进入提交、推送或整合流程。最终验收不通过时，不允许开始下一条路线。

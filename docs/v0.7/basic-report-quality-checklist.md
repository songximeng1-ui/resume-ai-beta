# V0.7 基础版报告质量检查清单

## 1. 核心验收线

一个普通应届生看完基础版报告后，能在 5 分钟内说出“我今天先做什么”，并且觉得这条建议和自己的材料有关。

这条标准优先于报告长度、模块完整度和表达精致度。基础版报告不追求像深度 AI 报告一样完整，而是要让用户能安全地开始下一步行动。

## 2. 基础版定位

基础版不是深度 AI 报告，不追求完整诊断；它的责任是保守、真实、可行动，并在 AI 深度报告失败或质量不稳时，仍然给用户一个不误导、能开始行动的兜底结果。

基础版必须明确展示为“基础版兜底报告”。不能因为 `source: real` 就让用户误以为这是完整深度 AI 报告。

基础版可以少，但不能假；可以保守，但不能空泛；可以只给一个小行动，但这个行动必须和用户当前材料有关。

## 3. 五个可检查维度

| 维度 | 通过标准 | 不通过标准 |
|---|---|---|
| 看得懂 | 普通应届生不需要理解产品术语，也能复述报告在说什么；句子短，动作具体。 | 大量出现“提升竞争力”“优化定位”“建立职业规划”等抽象表达，用户看完仍不知道含义。 |
| 知道今天做什么 | 报告突出 1 个今日主行动，用户能在 5 分钟内说出“我今天先做什么”。 | 只有长期建议，或多个并列任务抢主次，用户不知道先做哪个。 |
| 和我有关 | 报告引用用户输入中的具体动作、岗位、JD 要求、投递记录关键词或经历内容。 | 只引用 asset title，或只输出通用模板，看不出和用户材料的关系。 |
| 做得完 | 今日主行动能在 20-35 分钟内完成，有明确产出。 | 要求用户一次性做完整简历、完整职业规划、大项目、批量投递或长期任务。 |
| 不误导 | 保守、真实、不编造、不承诺结果，不评价用户本人。 | 出现 offer 承诺、录取预测、职业推荐、用户本人评价、伪造经历、夸大成果或焦虑式加量投递。 |

## 4. 四条路线的基础版要求

### `has_direction_resume_not_ready`

- 今日主行动：先选 1 段最稳的真实经历，补清对象、本人动作、工具、周期和交付物。
- 必须引用什么用户材料：用户真实经历中的具体动作、工具、项目或材料，例如“维护学生社群”“整理公众号推文素材”“设计问卷”“Excel 整理结果”。
- 禁止输出什么：完美简历、夸大经历、包装亮点、虚构成果、直接承诺生成完整简历。

### `target_job_fit`

- 今日主行动：逐条标出当前材料能证明哪些岗位要求，哪些证据不足，投递前先改哪一条简历表达。
- 必须引用什么用户材料：目标 JD 中的具体要求，以及用户真实经历中的动作或证据，例如岗位要求里的“用户反馈整理”和用户材料里的“记录用户反馈”。
- 禁止输出什么：保证 offer、一定进面、匹配度百分比、绝对能投或不能投、你不适合、录取预测。

### `applying_no_feedback`

- 今日主行动：整理最多 3 条投递记录，记录岗位、简历版本、投递时间、反馈状态和可疑线索。
- 必须引用什么用户材料：用户已有投递线索、简历版本、岗位类型、反馈状态，或可用于复盘的真实经历关键词。
- 禁止输出什么：你不行、简历很差、海投、每天投很多、盲目加量投递、把无反馈归因于用户本人。

### `no_direction`

- 今日主行动：先找 1 个真实可搜索的初级岗位样本。
- 必须引用什么用户材料：用户经历只能作为可探索方向的证据线索，例如某段经历里的动作、工具或项目；没有真实岗位样本前，不能把经历线索直接变成方向结论。
- 禁止输出什么：推荐职业、推荐你做、你最适合、AI 帮你选职业、你的职业方向是、未经岗位验证的方向结论。

## 5. 人工样本评分表

| 样本编号 | route | 用户材料摘要 | 用户是否能说出今日行动 | 报告是否引用用户真实材料 | 今日行动是否 20-35 分钟可完成 | 是否出现模板感/空泛感 | 是否出现伦理风险 | 总评：通过 / 有条件通过 / 不通过 |
|---|---|---|---|---|---|---|---|---|
| BR-001 | has_direction_resume_not_ready | 示例：有社群运营实习，但简历没写好 | 是 / 否 | 是 / 否 | 是 / 否 | 无 / 轻微 / 明显 | 无 / 有 | 通过 / 有条件通过 / 不通过 |
| BR-002 | target_job_fit | 示例：有目标 JD，想判断能不能投 | 是 / 否 | 是 / 否 | 是 / 否 | 无 / 轻微 / 明显 | 无 / 有 | 通过 / 有条件通过 / 不通过 |
| BR-003 | applying_no_feedback | 示例：投递多次但没有反馈 | 是 / 否 | 是 / 否 | 是 / 否 | 无 / 轻微 / 明显 | 无 / 有 | 通过 / 有条件通过 / 不通过 |
| BR-004 | no_direction | 示例：没有方向，只有零散经历 | 是 / 否 | 是 / 否 | 是 / 否 | 无 / 轻微 / 明显 | 无 / 有 | 通过 / 有条件通过 / 不通过 |

总评规则：

- 通过：5 个维度全部通过，且无伦理风险。
- 有条件通过：只有轻微模板感等小问题，但今日行动清楚、无伦理风险。
- 不通过：用户说不出今天做什么，或建议和材料无关，或出现伦理风险。

## 6. 自动测试 vs 人工验收边界

自动测试适合检查：

- `isBasic: true` 是否显示“基础版兜底报告”。
- 四条 route 是否都能生成基础版内容。
- 禁词，例如 offer 承诺、匹配度百分比、职业推荐、用户本人评价、盲目加量投递。
- 今日行动路线关键词，例如“先整理经历证据”“当前材料能证明岗位要求”“最多 3 条投递记录”“1 个真实可搜索的初级岗位样本”。
- 是否引用用户材料片段或具体关键词。
- 是否避免明显重复模板句，例如同一报告多处重复“执行、协作、整理、反馈”。

人工验收必须判断：

- 用户是否真的看得懂。
- 用户是否觉得报告和自己有关。
- 是否有模板感或敷衍感。
- 任务是否真的轻量，是否能在 20-35 分钟内完成。
- 语气是否现实温柔，是否既不打击用户，也不制造虚假承诺。

## 7. 额外约束

- “和我有关”不能只靠 asset title 假个性化；尽量引用用户输入中的具体动作、岗位、JD 要求或投递记录关键词。
- 基础版每次只突出 1 个今日主行动，其他建议只能作为辅助提醒，不能让用户不知道先做哪个。

## 8. V0.7 边界

本质量标准只服务 V0.7 基础版兜底报告修复，不进入 V0.8。

本分支不新增模型，不改变 DeepSeek 主模型 + Qwen fallback，不新增数据库、账号或后台，不修改 `V07JobRoute` 命名，不做职业推荐，不做自动投递、CRM、爬虫或支付。

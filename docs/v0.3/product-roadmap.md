# V0.3 产品路线图

## 产品定位

面向普通本科 / 双非 / 末流 211 准应届生、应届生的 AI 求职诊断工具。

一句话价值：帮用户把真实经历翻译成岗位语言，看清能投什么、怎么写简历、怎么准备面试、下一步补什么。

## 核心用户

- 大三 / 准应届生：经历不够完整，想提前规划实习、项目、技能和秋招筹码。
- 大四 / 应届生：已有简历或零散经历，想判断当前岗位匹配度、优化投递材料、准备面试。
- 典型状态：信心不足、经历表达弱、不了解岗位语言、容易把“没大厂实习”等同于“没价值”。

## V0.3 产品目标

让普通本科 / 双非 / 末流 211 的准应届生、应届生，无论有没有明确目标 JD，都能通过一次 AI 求职诊断，看清自己的真实经历筹码、适合方向、简历表达方式、面试风险和下一步补强路径。

## 双模式设计

### 无 JD 经历盘点

- 适合没有明确目标岗位、想先看自己能往哪里走的用户。
- 重点输出：经历资产、可迁移能力、相邻岗位方向、补强优先级、2-8 周行动路线。

### 有 JD 定制诊断

- 适合已经有目标岗位 JD、想判断能否投递的用户。
- 重点输出：JD 证据矩阵、投递判断、简历改写、面试追问、短期补强策略。

## 总用户流程

1. 选择阶段：准应届生 / 应届生。
2. 选择模式：无 JD 经历盘点 / 有 JD 定制诊断。
3. 输入简历文本与基础信息。
4. AI 结构化生成经历资产卡。
5. 用户确认真实经历、编辑或删除不适合内容。
6. AI 动态追问，补足任务、规模、工具、结果、复盘。
7. 无 JD 模式进入方向诊断；有 JD 模式进入 JD 证据匹配。
8. 生成诊断报告。
9. 用户反馈报告是否有用、是否愿意继续使用或付费。

## 核心功能模块

- 开始与模式选择。
- 简历输入与基础信息采集。
- 简历结构化与经历资产卡。
- 动态经历挖掘。
- 无 JD 方向诊断。
- 有 JD 证据匹配。
- 诊断报告生成。
- 安全提示与隐私删除。
- 用户反馈与匿名数据沉淀。

## 研发任务拆分

1. 全局流程整理：统一双模式路由、步骤状态、错误状态。
2. 开始页与输入页优化：降低填写压力，突出真实 AI 状态。
3. 经历资产卡升级：支持状态标签、缺口卡、确认真实、待核实。
4. 动态追问升级：结合经历与目标方向生成追问。
5. 有 JD 匹配页：输出岗位要求到用户证据的矩阵。
6. 无 JD 方向页：输出相邻岗位方向和补强优先级。
7. 报告页拆分：有 JD 报告与无 JD 报告分别组织。
8. 反馈数据：记录用户对报告有用性、准确性、行动意愿的反馈。
9. AI schema 与 prompt 维护：统一输出结构与安全红线。
10. 测试与验收：覆盖功能、AI 输出、安全、移动端和真实用户测试。

## 测试验收标准

- 用户能完整走通无 JD 和有 JD 两条流程。
- AI 结构化结果不伪造经历，缺失字段标记为待确认或缺口。
- 动态追问结合具体经历，不是固定模板。
- 有 JD 模式必须输出证据矩阵和投递判断。
- 无 JD 模式必须输出方向建议和补强路线。
- 报告必须包含简历表达、面试风险、行动计划和安全提示。
- 页面必须显示真实 AI / 演示模式 / 调用失败状态。
- 移动端基本可用，无关键按钮或文本遮挡。
- 用户可清除本次分析数据。

## 上线与反馈计划

- 第 1 轮：自己测试，重点验证流程和明显错误。
- 第 2 轮：3-5 个熟人测试，观察理解成本和卡点。
- 第 3 轮：10-20 个真实目标用户测试，收集报告有用性与行动意愿。
- 第 4 轮：20-50 人小范围付费意愿测试，验证真实需求强度。

## 分批开发 Prompt 策略

- 每个 prompt 只覆盖一个可验收功能块。
- 先修全局流程，再逐页推进，不一次性重构所有页面。
- 每个功能 prompt 都包含输入、输出、验收标准、安全限制。
- AI 输出 schema 与页面开发分开处理，避免前端和 prompt 同时失控。
- 每批开发后都运行构建、测试和手动流程检查。
